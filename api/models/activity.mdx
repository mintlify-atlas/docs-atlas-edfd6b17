## Overview

The Activity model tracks user actions related to venues, such as expressing interest, booking, and checking in. Activities are used to populate friend activity feeds, enabling social discovery features.

## Model Definition

<ResponseField name="id" type="string" required>
  Unique identifier for the activity.
</ResponseField>

<ResponseField name="user_id" type="string" required>
  ID of the user who performed the action. References the `users` table.
</ResponseField>

<ResponseField name="venue_id" type="string" required>
  ID of the venue the action relates to. References the `venues` table.
</ResponseField>

<ResponseField name="action" type="string" required>
  Type of action performed. Valid values:
  - `"interested"` - User expressed interest in the venue
  - `"booked"` - User booked or made a reservation at the venue
  - `"checked_in"` - User checked in at the venue
</ResponseField>

<ResponseField name="created_at" type="datetime" required>
  Timestamp when the activity occurred. Automatically set on creation.
</ResponseField>

## Validation Rules

- **id**: Required, must be a string (max 100 characters in database)
- **user_id**: Required, must reference an existing user
- **venue_id**: Required, must reference an existing venue
- **action**: Required, must be one of: `"interested"`, `"booked"`, `"checked_in"`
- **created_at**: Automatically set to current timestamp on creation

## Database Storage

Activities are stored in the `activities` table with the following characteristics:

- **Primary Key**: `id`
- **Foreign Keys**: 
  - `user_id` references `users.id` (CASCADE on delete)
  - `venue_id` references `venues.id` (CASCADE on delete)
- **Indexes**: 
  - `idx_activity_user` on `user_id` for efficient user activity queries
  - `idx_activity_venue` on `venue_id` for venue-specific activity queries
  - `idx_activity_created` on `created_at` for chronological sorting
- **Relationships**:
  - Many-to-one with users (via `user_id`)
  - Many-to-one with venues (via `venue_id`)

## Action Types

The `action` field supports three types of user activities:

### interested
Created when a user expresses interest in a venue. This is the most common activity type and triggers:
- Interest tracking for action item threshold detection
- Friend feed updates
- Potential action item creation when threshold is met

### booked
Created when a user books or makes a reservation at a venue. This indicates a higher level of commitment and:
- Appears in friend feeds with special styling
- May be used for venue popularity metrics

### checked_in
Created when a user physically checks in at a venue. This confirms actual venue visits and:
- Provides social proof in friend feeds
- Can be used for venue visit analytics
- May unlock special features or rewards

## Usage Notes

- Activities are automatically created by the system when users perform tracked actions
- The `created_at` timestamp is indexed for efficient chronological feed queries
- Cascade deletion ensures activities are removed when users or venues are deleted
- Activities are typically retrieved in reverse chronological order for feed display
- Friend feeds filter activities to show only actions from connected users
